後端架構視覺化說明

整潔架構 + FCIS + Decider + Event Sourcing

1 四個概念各管一件事

四個概念不是擇一，而是疊加使用，各負責不同面向。

整潔架構

管「分層」和「誰可以呼叫誰」
外層可以呼叫內層，反過來不行

FCIS

管「副作用隔離」
業務規則是純函數，不碰 DB / HTTP

Decider

管「函數簽名」
固定三元組：decide / evolve / initialState

Event Sourcing

管「怎麼存資料」
只存事件，狀態從事件重播算出來

關鍵規則：Domain 不 import 任何外層。它只接收值、回傳值，不知道 DB 和 HTTP 的存在。

3 Decider 三元組 — 每個業務操作的固定結構

每個操作都由三個部分組成，完美對應 Gherkin 測試的 Given / When / Then。

Given
evolve
「事件發生後，狀態變怎樣？」
evolve(state, 存款1000) → { balance: 1000 }
When
decide
「這個命令，目前能不能做？」
decide(state, 提款900) → { error: '餘額不足' }
Then
assert
「decide 的結果對不對？」
expect(result) .toEqual({ error: ... })

以「建立菜單項目 漢堡 $100」為例，資料怎麼流過每一層。

5 查詢流程 — 簡單直接

查詢不經過 Domain 和 Event Store，直接從 Read Model 讀取。

%%{init: {'theme': 'dark'}}%% sequenceDiagram participant U as 使用者 participant API as API 層 participant RM as Read Model U->>API: GET /menu/items API->>RM: getMenuItems() RM-->>API: [漢堡, 薯條, 可樂...] API-->>U: { success: true, items: [...] } Note over API,RM: 寫入走 Decider 全套流程
查詢直接讀 Read Model，超快

6 Event Sourcing vs 傳統 CRUD

Event Sourcing 存的是「發生了什麼事」，CRUD 存的是「現在長怎樣」。

傳統 CRUD

  • 直接改資料庫的「現在值」
  • UPDATE balance = 700
  • 歷史紀錄？不知道
  • 怎麼變成 700 的？不知道
vs

Event Sourcing

  • 只記錄每一筆「發生的事」
  • 存款 1000 → 提款 300
  • 完整歷史，可以審計
  • 重播就知道怎麼變成 700

7 為什麼選 ES — Codegen 轉換路徑

ES 是 Gherkin 和 CRUD 之間的中繼站，讓兩步轉換都是機械化操作。

結論：走上面那條路（Gherkin → ES → CRUD），兩步都可以讓 AI 自動完成。走下面直達路需要人工介入。所以 ES 是 codegen 的最佳中繼架構。
