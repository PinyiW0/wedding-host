前端工作流程視覺化說明

SDD + AI Codegen + TDD

1 核心理念 — 三個原則

上游 repo 產出規格，本專案 AI 生成程式碼，測試守護合約。每個產出都有明確的真相來源。

規格是唯一真相

AI 與操作者問答釐清 → .flow.md
手動放入本專案，驅動一切

合約先行

先定義型別和 API 合約
再寫測試合約
最後才寫 UI

骨架優先

UI 分階段建構
先搭骨架（路由、Layout）
再填細節（元件、頁面邏輯）

2 四大階段 — 從規格到綠燈

整個流程分四大階段依序執行，每個階段的產出是下一階段的輸入。

Phase A：API 合約

分析 .feature，建立型別定義、路由對照表、Mock 資料和 API 端點。是所有後續步驟的基礎建設。

/feature-to-api

Phase B：測試合約

從 .flow.md 生成 Playwright spec。定義每個 testid、互動模式和斷言預期。是 UI 的驗收標準。

/test e2e spec

Phase C：UI 實作

分 5 個 Phase 建構完整 UI — 主題 → 骨架 → Layout → 元件 → 頁面。為通過 spec 而建。

/feature-to-ui

Phase D：綠燈

跑 E2E 測試，找出失敗項目，修 UI/Mock/API 直到所有 spec 通過。

/test e2e green

8 Sync 模式 — 增量更新

Feature 變更時不需從頭跑。透過 content_hash 偵測變更，只更新受影響的部分。

全量模式（首次）

  • route-map.yaml 不存在
  • 從零建立所有產出
  • 掃描全部 .feature
  • 建立完整型別 + API + UI
vs

Sync 模式（迭代）

  • route-map.yaml 已存在
  • 比對 content_hash 找變更
  • 產出 sync-report.md
  • 只更新受影響的檔案

9 標準開發順序 — 走一遍完整流程

從放入規格檔到所有測試通過，每一步該下什麼指令。

Step 0 — 放入規格檔
從上游 repo 取得產出（EventStorming skill → .dsl.feature → AI 與操作者問答釐清 → .flow.md），手動放入 spec/gherkin-feature/ 和 spec/e2e-flows/
Step 1 — API 合約
分析 feature → 建立型別 + 路由對照 + Mock 資料 + API 端點
/feature-to-api
Step 2 — 測試合約
從 .flow.md 生成所有 Playwright spec
/test e2e setup ← 首次需要 /test e2e auto ← 自動偵測生成
Step 3 — UI 實作
五階段建構完整 UI — 主題 → 骨架 → Layout → 元件 → 頁面
/feature-to-ui ← Phase 1 開始 /feature-to-ui 5 {功能名} ← 逐頁實作
Step 4 — 綠燈
跑測試 → 修 UI → 重跑 → 直到所有 spec 通過
/test e2e green auto
